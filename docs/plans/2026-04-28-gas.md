# Gameplay Ability System (GAS) — Full Data-Driven Migration

## Problem

Everything that makes the game _play_ is hardcoded in C++:

| What | Where | Example |
|------|-------|---------|
| All buyable items (weapons, bombs, items) | `world.cpp:1380–1395` | `new BuyableItem(BUY_LASER, "Laser", 150, 300, ...)` |
| Weapon projectile damage | `blasterprojectile.cpp`, `laserprojectile.cpp`, etc. | `healthdamage = 40; shielddamage = 4;` |
| Agency upgrade stat caps | `user.h:37–42` | `static const Uint8 maxendurance = 5;` |
| Agency upgrade multipliers | `player.cpp:3487–3490` | `maxshield += agency.shield * 20` (magic 20) |
| Player base stats | `player.cpp:67–70` | `maxhealth = 100; maxshield = 100;` |
| Item behavior switch | `game.cpp:2274` | `switch(buyableitem->id) { case BUY_LASER: ... }` |
| Agency-specific item restrictions | `world.cpp` | `Team::NOXIS`, `Team::BLACKROSE` inline |
| Guard / NPC stats | `guard.cpp`, `robot.cpp` | hardcoded health, speed, detection range |

Rebalancing anything requires a C++ recompile and a new client build pushed to all players.

## Goal

Move all game mechanics data into JSON files under `shared/assets/gas/`, loaded at
startup. No recompile needed to rebalance — edit JSON, restart server (or hot-reload on
next map load).

Full scope:
- **Agencies** — base stats, upgrade caps, upgrade cost table, per-stat multipliers
- **Items / weapons** — prices, descriptions, tech requirements, agency restrictions, effects
- **Weapon damage** — shield damage, health damage, projectile behaviour per weapon type
- **Player** — base health, base shield, spawn stats
- **Enemies** — guard health, speed, detection range, alert thresholds, patrol behaviour params
- **Abilities** — cooldown, credit cost, effect type, duration, stacking rules
- **Game objects** — heal machine rates, credit machine payouts, pickup values

## Phase 1 — JSON Schema + loader infrastructure

### Files in `shared/assets/gas/`
```
agencies.json      — 5 agency definitions
items.json         — all BuyableItem entries
weapons.json       — projectile/weapon damage & behaviour
player.json        — player base stats & spawn config
enemies.json       — guard, robot, civilian NPC stats
abilities.json     — active ability definitions
gameobjects.json   — heal rates, credit payouts, pickup values
```

### C++ loader (`src/gas/`)
- `GASLoader` — singleton, loads all JSON at startup from `ASSETS_DIR/gas/`
- Exposes typed structs: `AgencyDef`, `ItemDef`, `WeaponDef`, `EnemyDef`, `AbilityDef`
- Hot-reload: `GASLoader::Reload()` safe to call between map loads
- Falls back to compiled-in defaults if file missing (backwards compat during migration)

## Phase 2 — Agency & player migration

Replace hardcoded values in:
- `user.h` — `static const Uint8 maxendurance = 5` → loaded from `AgencyDef::upgradeCaps`
- `player.cpp:3487–3490` — `shield * 20` magic → `AgencyDef::upgradeMultiplier.shield`
- `player.cpp:67–70` — base health/shield → `PlayerDef::baseHealth`, `PlayerDef::baseShield`
- `creditmachine.cpp` — upgrade cost table → `AgencyDef::upgradeCosts[]`

`agencies.json` shape:
```json
{
  "agencies": [
    {
      "id": 0,
      "name": "Noxis",
      "color": [220, 60, 60],
      "baseStats": { "health": 100, "shield": 100 },
      "upgradeCaps": { "endurance": 5, "shield": 5, "jetpack": 5, "techslots": 8, "hacking": 5, "contacts": 5 },
      "upgradeMultipliers": { "endurance": 20, "shield": 20 },
      "upgradeCosts": [100, 200, 400, 800, 1600]
    }
  ]
}
```

## Phase 3 — Items & weapons migration

Replace `world.cpp:1380–1395` block with JSON loader:
- `World::LoadBuyableItems()` reads `items.json` → populates `world.buyableitems`
- Remove `BUY_*` enum values and inline constructor calls

Replace hardcoded weapon damage in each `*projectile.cpp`:
- Projectile constructors call `GASLoader::Get().GetWeaponDef(type)` for damage values
- `weapons.json` defines `shieldDamage`, `healthDamage`, `speed`, `lifetime` per weapon

`items.json` shape:
```json
{
  "items": [
    {
      "id": "laser", "name": "Laser", "price": 150, "repairPrice": 300,
      "spriteBank": 97, "spriteIndex": 16,
      "techChoice": 1, "techSlots": 1,
      "agencyRestriction": null,
      "description": "Immediately after shield technology..."
    }
  ]
}
```

`weapons.json` shape:
```json
{
  "weapons": [
    { "id": "laser",  "shieldDamage": 60, "healthDamage": 10, "speed": 12, "lifetime": 80 },
    { "id": "rocket", "shieldDamage": 25, "healthDamage": 75, "speed": 8,  "lifetime": 120 },
    { "id": "blaster","shieldDamage": 4,  "healthDamage": 40, "speed": 10, "lifetime": 60 }
  ]
}
```

## Phase 6 — GAS Tool (admin UI, local-first, git-tracked)

The GAS Tool lives in the admin UI at `/gas` — same pattern as Actor Editor, Sound Studio, and Map Designer.

**No server API, no server persistence.** The user opens JSON files from their local `shared/assets/gas/` checkout, edits them inside the tool, and saves them back to disk. The JSON files are committed to git and shipped as part of each tagged client/server release. This guarantees full history, code review, and rollback via git.

### UX flow

1. User opens `/gas` — shown an empty-state gate (same style as Actor Editor / Designer)
2. User clicks **OPEN GAS FOLDER** → browser folder picker points at their local `shared/assets/gas/`
3. Tool reads all 7 JSON files from the selected folder (agencies, items, weapons, player, enemies, abilities, gameobjects)
4. User edits data in tabbed editors
5. User clicks **SAVE** on a file → browser `showSaveFilePicker` / download writes the modified JSON back to the same path
6. User commits the changed JSON files to git, tags a release — game picks up the new data on next launch

### Sub-tools inside `/gas` (tabs or sidebar nav)

| Sub-tool | JSON file | What you can edit |
|----------|-----------|-------------------|
| Agencies | `agencies.json` | Base stats, upgrade caps, cost table, multipliers per agency |
| Items | `items.json` | Add/edit/remove buyable items — price, repair, sprite, tech slots, agency restriction, description |
| Weapons | `weapons.json` | Shield/health damage, projectile speed, lifetime per weapon |
| Player | `player.json` | Base health, shield, spawn config |
| Enemies | `enemies.json` | Guard/robot/civilian health, speed, detection, patrol params |
| Abilities | `abilities.json` | Credit cost, cooldown, effectType, effectParams per ability |
| Game Objects | `gameobjects.json` | Heal machine rates, credit payouts, pickup values |

### No admin API needed for GAS

The GAS Tool reads and writes files entirely in the browser (via `FileSystemDirectoryHandle` / `File` / `Blob` download). No new server routes are required. `services/admin-api` is not touched for GAS.

## Migration strategy

### 1:1 parity requirement

**The migration must not change any game behaviour.** Every value extracted into JSON must exactly match the current hardcoded C++ value. No balance tweaks, no new abilities, no feature additions — the game before and after each phase must play identically. Rebalancing is a separate, future concern enabled by the GAS infrastructure; it is out of scope for this migration.

Each phase PR must include a diff showing:
- The C++ value that was removed
- The JSON value that replaced it (same number)

1. Loader reads JSON first; falls back to C++ compiled-in values if file absent
2. Migrate one category at a time (agencies → items → weapons → enemies → abilities)
3. Each phase ships independently — game stays playable throughout
4. Old hardcoded paths deleted only after JSON equivalents are confirmed working in production
5. **All JSON changes flow through git** — no out-of-band edits via a server API, ensuring every balance change is traceable, reviewable, and rolls back cleanly
