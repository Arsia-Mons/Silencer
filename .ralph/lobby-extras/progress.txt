Three concrete textbox-content patterns confirmed: bottom-to-top textbox = `for i ∈ 0..N-1: DrawText(x, y_bottom - (N-1-i)*lineheight)` (chat scrollback at lineheight=11, fontwidth=6); top-to-bottom textbox = `for i: DrawText(x, y_top + i*lineheight)` (presence at lineheight=11); SelectBox rows = same as top-to-bottom textbox but with lineheight=14 and a small ~3px left padding from the bbox x to clear the chrome inner edge. Reference dump ordering is authoritative when prompt-prose ordering disagrees (presence list shows Ember-above-Halcyon despite RALPH.md listing Halcyon first) — always visually check the reference before trusting prose. Whole-screen pixelwise diff RISES when populating interiors (1.23% empty → 3.89% populated) because effectcolor-disagreeing colored glyphs replace mostly-matching black-empty interiors; this is expected and the structural metric (named objects at named anchors) is the actual gate, not whole-screen %. Reference's TextBox widget word-wraps long lines (Quill/Krieg in presence; Solace's chat); candidate doesn't model word-wrap and renders single-line — non-gating since names/messages are structurally present at correct anchors.
- LOBBY-derivative modal pattern (E1..E3): the modal screens that "replace GameSelectInterface" are RunDumpLobby with a single bbox-scoped substitution. Spec coordinates with vague placeholders like "(right column) (~190)" require visual interpretation against the reference: Select Maps label is actually at y~88 (top of right column above the list), not y=190.
