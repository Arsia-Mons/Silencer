- Empty-data → populated-data is purely additive DrawText: every populated content slot is already a "spec'd structural anchor" comment from the empty-data Ralph, so filling them in is mechanical (bank + advance + literal anchor coords from the spec table; no new banks, no new sprites, no new codec branches). RunDumpLobby's E0 delta is six contiguous DrawText blocks layered on top of validated chrome. Three concrete textbox-content patterns confirmed: bottom-to-top textbox = `for i ∈ 0..N-1: DrawText(x, y_bottom - (N-1-i)*lineheight)` (chat scrollback at lineheight=11, fontwidth=6); top-to-bottom textbox = `for i: DrawText(x, y_top + i*lineheight)` (presence at lineheight=11); SelectBox rows = same as top-to-bottom textbox but with lineheight=14 and a small ~3px left padding from the bbox x to clear the chrome inner edge. Reference dump ordering is authoritative when prompt-prose ordering disagrees (presence list shows Ember-above-Halcyon despite RALPH.md listing Halcyon first) — always visually check the reference before trusting prose. Whole-screen pixelwise diff RISES when populating interiors (1.23% empty → 3.89% populated) because effectcolor-disagreeing colored glyphs replace mostly-matching black-empty interiors; this is expected and the structural metric (named objects at named anchors) is the actual gate, not whole-screen %. Reference's TextBox widget word-wraps long lines (Quill/Krieg in presence; Solace's chat); candidate doesn't model word-wrap and renders single-line — non-gating since names/messages are structurally present at correct anchors.

- **Learnings:**
  - **The empty-data → populated-data jump is mostly DrawText calls,
    not new sprites or codec branches.** Every populated content slot
    was already a "spec'd structural anchor" in the I0/I1/I2 comments
    from the prior Ralph; filling them in is mechanical: pick the bank
    + advance from the spec table, pick the literal anchor coords,
    DrawText. No new bank loads, no new chrome sprites — the whole E0
    delta is six contiguous DrawText blocks. This pattern will repeat
    for the modal/state screens E1..E5 — where they reuse LOBBY
    chrome, they reuse this whole render path; where they replace
    GameSelectInterface (E1..E3), the substitution is bbox-scoped.
  - **Reference dump ordering can disagree with prompt list ordering.**
    RALPH.md lists "In Lobby: Halcyon, Ember, Solace, Vector, demo"
    but the reference shows "Ember, Halcyon, Solace, Vector, demo".
    Always check the reference visually before trusting prompt-prose
    ordering — the rendering order is whatever the engine's textbox
    insert sequence produced, not the prompt-author's recall order.
  - **Pixelwise diff RISES when populated content lands** (1.23% →
    3.89%) — counterintuitive but expected. The empty-data candidate
    had MOSTLY-MATCHING black-empty interiors that just happened to
    coincide with the reference's black-empty interiors; populating
    those interiors with colored glyphs introduces effectcolor and
    word-wrap disagreement pixels. The structural metric (named
    objects present at named anchors) is the right gate; whole-screen
    pixel diff is dominated by effectcolor-modeling gaps.
