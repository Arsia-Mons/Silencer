- Screen-switch extension recipe carries forward unchanged from D0 / A0 / O0 / C0: one new `if (screen_str == "lobby_connect")` branch in `main()` plus a sibling `RunDumpLobbyConnect(...)` function. Zero edits to prior Run*; bytewise diffs vs all five prior reference dumps stay at exact baseline (main 2136 / options 1758 / audio 1035 / display 2211 / controls 3729). The "additive screen" pattern continues to hold across the menu→lobby palette boundary.
- Sub-palette switching is per-`RunDump*`, not per-screen-state: each screen-builder owns a `constexpr int kSubX = N;` constant and passes it to `WritePPM(fb, palette, kSubX)`. Menu screens use `kSubMenu = 1`; LOBBYCONNECT introduces `kSubLobby = 2`. No rework of `WritePPM` or `Palette` was needed — the active-sub argument was always per-call, not global. The "first non-menu palette" gate (L0/L1) is therefore satisfied by a one-line change in the new screen-builder, with no shared-infra churn.
- Sub-palette 2 idx 0 = (0, 0, 0) black, confirmed empirically by dumping a Clear(0) FB resolved through sub-palette 2 — the first 12 RGB bytes of the dump payload are all 0x00. This means `Clear(0)` produces a black panel-exterior background "for free" under sub-palette 2, with no need for an explicit black-fill widget. The same holds for sub-palette 1 (where main_menu's bg_plate-less zones would also be black if any were exposed; main_menu has none).
- LOBBYCONNECT bank set planned ahead at L0 commit: `{7, 133, 134, 135}`. Bank 7 (panel idx 2 + likely B52x21 chrome), bank 133 (textbox + textinput font), bank 134 (Username/Password labels font, w=9), bank 135 (button-label font, by analogy with B196x33/B220x33/B112x33). NO bank 6 (no starfield bg, no bg-plate idx 0) — the panel sprite carries the chrome and the exterior is `Clear(0)` per spec. Bank set is loaded at L0 even though L0 doesn't render any of them yet, to avoid bank-list churn in L1+. **Confirmed at L2:** the bank-7 idx-2 sprite carries ALL panel chrome — outer bordered rectangle, inner textbox border (with right-side scrollbar lane baked in), horizontal divider, Username/Password input-field wells, AND the bottom button-row well outlines for Login/Cancel. Same single-blit pattern as CONTROLS' bank-7 idx-7 frame: one `BlitSprite(fb, sprites.Get(7, 2), 0, 0, nullptr)` produces the entire form scaffold.
- **PALETTE.BIN actual layout is `4 + s * (768 + 4)`** — i.e. 4-byte file prefix + 11 records of [4-byte sub-header, 768-byte color table]. The main-menu Ralph's pattern claim of `8 + s*768` was a coincidence: at s=1 both formulas give offset 776, so every prior screen (which all use sub-palette 1) rendered correctly under the wrong formula. They diverge for s != 1 — sub-palette 2 (lobby) sits 4 bytes farther per sub-block, so the empirical formula was reading 4 bytes early and pulling garbage. Correcting to the spec formula leaves sub-palette 1 byte-identical (zero regression on main_menu / options / options_audio / options_display / options_controls dumps — all five at exact baselines: 2136 / 1758 / 1035 / 2211 / 3729) while shifting sub-palettes 0 and 2..10 to their correct offsets. **Lesson:** whenever a "spec disagrees with empirical" is recorded, verify the empirical against MORE THAN ONE data point — sub-palette 1 was the only one tested in main-menu Ralph P0; sub-palette 2 disagreed silently because no prior screen exercised it.
- Verification mode for palette correctness gates does NOT require visual A/B: when the candidate's first non-`Clear(0)` render under the new sub-palette is gated separately (here, panel sprite is L2's job), L1's "color tones match" check is a table-lookup audit. Sample reference dump's dominant non-black colors → confirm each appears in `sub_palette[N]` post-fix. For this Ralph: top-15 full-screen reference colors all present in sub-palette 2 (15/15 hit rate), confirming the fix is decisive even though the candidate dump is still entirely black at the L1 gate.
