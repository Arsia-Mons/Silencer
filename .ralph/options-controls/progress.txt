  starfield bleed where no widgets occlude it.
- Spec gaps encountered (CONFIRMED + NEW):
  - **Bank 7 idx 7 is a single full-screen tile-mode RLE sprite**, not a
    composite of corner sprites. The spec's "may use tile-mode RLE codec"
    speculation is confirmed; no new codec work needed.
  - **The scrollbar TRACK is part of the frame-panel sprite**, not a
    separate widget. This contradicts the strict reading of the spec /
    archive:1080-1130 / `prd.json` C4 ("res_index=9 in the scrollbar bank
    is the track"). For OPTIONSCONTROLS rendering purposes, idx 9 is
    redundant — the only widget-rendered scrollbar element is the thumb
    (idx 10). The C4 spec should be updated to reflect: "thumb (idx 10)
    only — track is baked into the frame panel."
  - **`widget-frame.md` should be authored** documenting bank 7 idx 7 as
    a full-screen composite frame with baked title-plate cutout and
    scrollbar-track decoration. This is more useful framing than treating
    it as a generic "bordered panel" since the title-plate cutout
    constrains where the title overlay can go.
- **Learnings:**
  - The spec's framing (frame panel + separate scrollbar widget) was
    structurally correct but compositionally wrong — at the sprite level,
    the engine packs the "frame chrome" (border + title plate + scrollbar
    track) into one composite asset and renders only the dynamic parts
    (title text, form widgets, scrollbar thumb) on top. This is a useful
    pattern: when in doubt about whether decorative geometry is
    sprite-baked vs. widget-composed, blit the candidate sprite alone
    and inspect — the answer is usually visible at first sight.
  - Bank 7 was loaded at C0 but not visually exercised; C1's payload is
    therefore "do nothing new with assets, just blit the already-loaded
    sprite." The decoupling of bank-loading (C0) from bank-rendering
    (C1) caught no codec issues — but DID reveal the scrollbar-track
    pattern above, which is a downstream simplification for C4.
  - The visual gate (eyeballing the PNG) is decisive at C1, not the
    bytewise diff — 107K bytes of diff still represented a PASS because
    every diffed pixel is in a region intentionally blank at this iter.
    Pattern: "diff size ≠ correctness when missing widgets account for
    the diff."
