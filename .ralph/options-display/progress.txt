## 2026-04-26T15:08:00Z — iter 2 (D1 PASS)
- Filled in `RunDumpOptionsDisplay`'s composition body in
  `shared/design/sdl3/src/main.cpp` with all 7 spec objects from
  docs/design/screen-options-display.md: title `DrawText` ("Display
  Options", bank 135 advance 12, centered y=14); two B220x33 toggle
  buttons via a `rows` array indexed `i ∈ {0,1}` with anchors
  `(100, 50 + i*53)` and labels Fullscreen / Smooth Scaling; per-row
  off+on half-pills (bank 6 idx 12 at x=420, idx 15 at x=450) at
  literal y=137+i*53; Save/Cancel B196x33 at `(-200, 117)/(20, 117)`.
  Reused B220x33 width=220 / advance=11 / yoff=8 from `RunDumpOptionsAudio`,
  the validated audio-screen template.
- Build clean, dump 921615 bytes (canonical PPM size).
- Bytewise diff vs `/tmp/real_optionsdisplay_dump.ppm`: **2,211 bytes
  ≈ 0.24%** — under the empirically-calibrated 0.25% F0 threshold and
  within the same envelope as main-menu (0.23%) / options (0.19%) /
  options-audio (0.11%) baselines. Higher than audio's 0.11% because
  the screen has 2 toggle rows + 4 indicator pills + 2 longer button
  labels (Fullscreen, Smooth Scaling) — more glyph-aliased surface.
- Regression checks (all clean): main 2136, options 1758, audio 1035
  — exact matches to F0/O4/A4 baselines.
- Visual A/B (`/tmp/sdl3_dump/screen_00.png` vs
  `/tmp/real_optionsdisplay_dump.png`): indistinguishable to the
  casual reader. Title centered at top, both toggle rows at coincident
  y, both rows show dim-left-half + bright-right-half pills (Mars
  visible behind/below them), Save and Cancel at the bottom.
- **Pattern:** the OPTIONSAUDIO template is a near-perfect base for
  any "title + N toggle rows + Save/Cancel" screen — only the row
  count, row stride, label texts, and indicator y-base change. Future
  toggle screens (controls 6-row, etc.) should clone-and-loop the
  same body rather than refactor the audio version.
- **Pattern:** for indicator pills the spec's literal y=137 is engine-
  raw and adds the row stride (+53) on top, NOT relative to button
  anchor y. The button anchor (50+i*53) and indicator y (137+i*53)
  drift apart by sprite-offset arithmetic per the spec's "indicator-y
  is overlay-relative" note — both render at the visually-same row
  because B220 chrome offset_y is large negative.
- **Pattern:** the residue grows roughly linearly in glyph-aliased
  pixel count — main menu (4 short labels) ≈ 0.23%, options (4 short
  labels) 0.19%, audio (1 short label "Music" + 1 title) 0.11%, display
  (2 longer labels Fullscreen/Smooth Scaling + title + Save/Cancel) 0.24%.
  Anti-aliasing palette-index drift on glyph edges is the irreducible
  floor — quoted "Smooth Scaling" alone has 14 glyphs vs Music's 5.

## 2026-04-26T15:12:00Z — iter 3 (D2 PASS)
- Free-flip verification of the +53 row stride. D1's `RunDumpOptionsDisplay`
  already enumerates rows via `rows[i]` with `kToggleY0=50`, `kPillY0=137`,

  and the indicator pill literal coords `(420, 137+i*53)` / `(450, 137+i*53)`
  are stride-correct by construction. D2 is a verification-only flip.
- Build clean, dump 921615 bytes. Bytewise diff vs reference: **2,211 bytes
  ≈ 0.24%** — identical to D1 baseline (RNG-free determinism preserved).
- Decisive per-region pixel diff (sips-converted candidate PNG vs
  `/tmp/real_optionsdisplay_dump.png`):
  - row1 pill off (x∈[420,440] y∈[137,170], 660 px): **0 diff** (525==525 nz)
  - row1 pill on  (x∈[450,470] y∈[137,170], 660 px): **0 diff** (660==660 nz)
  - row2 pill off (x∈[420,440] y∈[190,223], 660 px): **0 diff** (659==659 nz)
  - row2 pill on  (x∈[450,470] y∈[190,223], 660 px): **0 diff** (660==660 nz)
  - both rows pill bbox (x∈[420,470] y∈[137,223], 4300 px): **0 diff** (4082==4082 nz)
  Pixel-equality across the full row1+row2 pill bounding box pins the
  +53 stride exactly — drift in either anchor or stride would scatter
  diff pixels into the inter-row gap, so a stride-of-anything-else
  cannot produce 0 diff over 4300 px.
- B220 button bands per row (x∈[0,640]):
  - row1 (y∈[136,170], 21760 px): 263 diff = 1.21% — nz_c=15847 == nz_r=15847
  - row2 (y∈[189,223], 21760 px): 412 diff = 1.89% — nz_c=16898 == nz_r=16898
  Identical opacity counts on both rows confirms glyph chrome and label
  text are pixel-coincident at the +53 stride; the diff is the same
  palette-index anti-aliasing nit shared with main-menu, options,
  options-audio (same shape: equal nz, palette drift on glyph edges).
- Regression checks (all clean): main_menu 2,136 (matches F0), options
  1,758 (matches O4), options_audio 1,035 (matches A4). No regression.
- Visual A/B (`/tmp/sdl3_dump/screen_00.png` vs reference): both rows
  fully coincident — Fullscreen and Smooth Scaling labels at correct
  vertical offsets, both off/on pill pairs at correct y, inter-row gap
  identical to reference. Indistinguishable to a casual reader.
- Files touched: `.ralph/options-display/prd.json` (D2 → passes:true),
  `.ralph/options-display/progress.txt` (this entry + one Codebase
  Patterns line). No source code changes.
- **Pattern:** for multi-row toggle screens, the decisive stride
  evidence is per-region pixel-equality on the *combined* row1+row2
  bbox of an unambiguous fixed-content element (here: the pills, which
  are pure sprite blits with no glyph aliasing). Pill bboxes are tiny
  (660 px each), nz-saturated, and palette-stable — 0 diff across the
  combined bbox is a stride-correctness proof that doesn't require
  arithmetic on per-row offsets. Future toggle-screen Ralphs (controls
  6-row, etc.) should reuse this pill-bbox technique as the stride
  gate, not button-band diffs (which carry glyph-aliasing residue).
